# phc_probe — PTP Hardware Clock Diagnostic Tool

## Overview

`phc_probe` compares hardware RX timestamps (`ts[2]`, generated by the ENA PTP Hardware Clock at the NIC) with software/kernel RX timestamps (`ts[0]`, generated by the ENA driver using `CLOCK_REALTIME`) for UDP packets received on an EC2 instance.

It is a two-host test: a sender on one machine sends UDP packets to a receiver on an EC2 instance with ENA hardware timestamping enabled. The receiver extracts both timestamp types from the kernel's `SO_TIMESTAMPING` ancillary data and displays them side by side.

**Why two hosts?** Loopback packets (sent and received on the same host) bypass the NIC entirely and never get hardware timestamps — `ts[2]` will always be zero. Packets must traverse the physical network to be timestamped by the ENA hardware.

## Scripts

| Script | Where to run | Privileges | Description |
|--------|-------------|------------|-------------|
| `compare_timestamps.sh` | Either host | Varies | Wrapper that dispatches to `ts_receiver.py` or `ts_sender.py` |
| `ts_receiver.py` | EC2 instance (receiver) | `sudo` required | Listens for UDP packets, extracts HW and SW timestamps, displays comparison |
| `ts_sender.py` | Any host (sender) | No privileges needed | Sends UDP packets with embedded sequence numbers and TX timestamps |

## Prerequisites

**Receiver host (EC2 instance):**
- EC2 instance with ENA driver supporting PTP Hardware Clock (PHC)
- PHC enabled: `cat /sys/module/ena/parameters/phc_enable` should return `1`
- Hardware timestamping verified: `ethtool -T <interface>` should show `hardware-receive` capability
- Python 3.9+
- Root access (for `SO_TIMESTAMPING` and `SIOCSHWTSTAMP` ioctl)

**Sender host:**
- Any Linux/macOS host with network connectivity to the receiver
- Python 3.6+
- No special drivers, privileges, or hardware needed

## Usage

### Using the wrapper script

On the receiver (EC2 with HW timestamps):
```bash
sudo bash scripts/compare_timestamps.sh server [interface] [--port PORT] [--count N]

# Example:
sudo bash scripts/compare_timestamps.sh server ens5 --port 9999 --count 20
```

On the sender (any host):
```bash
bash scripts/compare_timestamps.sh client <receiver-ip> [--port PORT] [--count N] [--interval MS]

# Example:
bash scripts/compare_timestamps.sh client 10.0.1.100 --port 9999 --count 20
```

### Using the Python scripts directly

On the receiver:
```bash
sudo python3 scripts/ts_receiver.py <interface> [--port PORT] [--count N]

# Example:
sudo python3 scripts/ts_receiver.py ens5 --port 9999 --count 20
```

On the sender:
```bash
python3 scripts/ts_sender.py <receiver-ip> [--port PORT] [--count N] [--interval MS]

# Example:
python3 scripts/ts_sender.py 10.0.1.100 --port 9999 --count 20 --interval 100
```

## Command-Line Flags

### ts_receiver.py

| Flag | Default | Description |
|------|---------|-------------|
| `interface` (positional) | — | Network interface name (e.g. `ens5`, `enp40s0`) |
| `--port PORT` | 9999 | UDP port to listen on |
| `--count N` | 10 | Number of packets to capture |
| `--live` | off | Live rolling ASCII graph of HW-SW delta (last 60s, runs until Ctrl+C) |

### ts_sender.py

| Flag | Default | Description |
|------|---------|-------------|
| `receiver_ip` (positional) | — | IP address of the receiver host |
| `--port PORT` | 9999 | UDP port to send to |
| `--count N` | 10 | Number of packets to send |
| `--interval MS` | 100 | Interval between packets in milliseconds |

## Understanding the HW-SW Delta

The delta between `ts[2]` (hardware) and `ts[0]` (software) reveals how much latency the kernel's network stack adds between the moment a packet arrives at the NIC wire and when the kernel delivers it to software.

```
wire arrival ──► NIC stamps ts[2] ──► IRQ ──► driver ──► kernel network stack ──► kernel stamps ts[0]
                 (hardware clock)        ◄─── this gap is the HW-SW delta ───►    (CLOCK_REALTIME)
```

**What the delta tells you:**
- **Stable, small negative delta** (ts[2] < ts[0], e.g. -20 to -30 us): Normal operation. The NIC stamps the packet at wire time, the kernel stamps it later after IRQ delivery and protocol processing. The magnitude is your kernel RX overhead.
- **Large or variable delta** (hundreds of microseconds, high jitter): Indicates kernel scheduling delays — IRQ coalescing, CPU contention, C-state wake-up latency, or interrupt affinity misconfiguration. Use the live graph to correlate spikes with system events.

## Packet Format

The sender transmits 12-byte UDP packets with the following layout:

```
Offset  Size  Field            Encoding
0       4     seq_num          uint32_t, network byte order
4       8     tx_app_timestamp uint64_t, network byte order, nanoseconds
```

## How It Works

1. The receiver creates a UDP socket and enables `SO_TIMESTAMPING` with flags for both hardware and software RX timestamps
2. The receiver calls `SIOCSHWTSTAMP` ioctl to enable hardware timestamping on the NIC (sets `rx_filter=HWTSTAMP_FILTER_ALL`)
3. The sender sends UDP packets with embedded sequence numbers and TX timestamps
4. The receiver calls `recvmsg()` which returns the packet data plus ancillary data containing a `scm_timestamping64` structure:
   - `ts[0]` — kernel software RX timestamp (from `CLOCK_REALTIME`)
   - `ts[1]` — deprecated/unused
   - `ts[2]` — hardware RX timestamp (from ENA PTP Hardware Clock)
5. The receiver displays both timestamps and computes the delta between them

## Troubleshooting

**No hardware timestamps (ts[2] always zero):**
- Packets must come from a REMOTE host — loopback bypasses the NIC
- Check PHC is enabled: `cat /sys/module/ena/parameters/phc_enable` (should be `1`)
- Check NIC capabilities: `ethtool -T <interface>` (should show `hardware-receive`)
- Check HW timestamping state: `cat /sys/bus/pci/devices/<pci-addr>/hw_packet_timestamping_state`
- Ensure running as root (`sudo`)

**Socket option errors:**
- `SO_TIMESTAMPING_NEW` (option 65) may not be supported on older kernels or under OpenOnload — the receiver falls back to `SO_TIMESTAMPING` (option 37) automatically

**Timeout waiting for packets:**
- Verify sender is targeting the correct IP and port
- Check security groups / firewall rules allow UDP traffic on the specified port
- Verify the receiver interface has an IP address: `ip addr show <interface>`
